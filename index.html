Introduce to test coverage

Why we need test coverage?

For software quality, we always write tests to ensure our program can execute correctly

A JavaScript unit tests example

If I have written a lof of test cases, how can I know what parts of the system functionality has been tested?


  • Module A
    • contructor() tested: ✔
    • initialize() tested: ✔
    • getView() tested: ✔
    • getModel() tested: ✘
    • destory() tested: ✔
  • Module B
    • getValue() tested: ✔

Web application

Nowadays, web is going to become more and more large and complicated


Gmail is a very good example

Web application composed by a lot of modules or third-party libraries


Webpage download many scripts or resources in background

Issue

  • Too many scripts download will cause network slow down

  • User always not use every functionality on website

  • Some libraries are so huge that will stuck webpage on loading

  • Some libraries are not very necessary when first time load

For reducing bandwidth and make better user experience, many approachs have been proposed to address these issues.

Script Loading - XHR


var xhr = new XMLHttpRequest();
xhr.onload = function(responseScript) {
	// execute response script 
	eval(responseScript);
};
xhr.open('GET', 'URL');
xhr.send();

Famous script loading API such as RequireJS, CommonJS use some syntax like:
