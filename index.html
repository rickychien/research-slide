Test coverage

Why we need test coverage?

For software quality, we always write tests to ensure our program can execute correctly

A JavaScript unit tests example

If I have written a lof of test cases, how can I know what parts of the system functionality has been tested?


Is possible to have a statistics like...


  • Module A
    • contructor() has been tested: ✔
    • initialize() has been tested: ✔
    • getView() has been tested: ✔
    • getModel() has been tested: ✘
    • destory() has been tested: ✔
  • Module B
    • getValue() has been tested: ✔
    • connect() has been tested: ✘

That's it! Test coverage tool do it for us


It reports what functionalitys have been tested

Blanket.js - JavaScript test coverage tool

  • Nowadays, web is going to become more complicated

  • Client (Browser) can take more job for server

  • Just like a desktop application but written by web technique


Gmail work just like a desktop application

To build a complicated web application


  • Developer wants to distribute JavaScript files to modules

  • Web page is composed by modules or third-party libraries


    • Web application is different with desktop application here


Web application load resources in background, but desktop application doesn't do this

New issues

  • Too many scripts download will cause network slow down

  • User always not use every functionality on website

  • Some libraries are so huge that will stuck webpage on loading

  • Also some libraries are unnecessary when first time load

Solutions by script loading

  • Preloading

  • Load script on demand

  • Load script through async or defer

Load script on demand or through async or defer can be lazy loading approach. When these lazy loading technique meet with test coverage tool, new problem will arise.

Test coverage with lazy loading

Zero coverage


  • Take firefox OS Gaia for an example. Email app has 27 source scripts, but it just show one file on coverage report

Analyze the reasons


Because coverage tool only can detect script in static


Take the blanket.js as an example


<script src="path/script.js" data-cover></script>

It detects "data-cover" attribute and instruments it's coverage

 


<script src="blanket.js" data-cover-only="src/"></script>

Or specify "data-cover-only" when loading blanket.js

Analyze the reasons


  • Web coverage tools only setup instrument scripts on beginning

  • Load script dynamically through javascript is so hard to detect


var xhr = new XMLHttpRequest();
xhr.onload = function (responseScript) {
  // Execute response script
  eval(responseScript);
};
xhr.open('GET', 'URL');
xhr.send();

Ajax is a dynamic way to load scripts

Script loading approachs

Demonstrate the web script loading approach

Script Loading - HTML Script



<script src="path/script.js"></script>

or


<script>
  // Script content
</script>

Script Loading - HTML Script (async/defer)



<script src="path/script.js" async></script>

or


<script src="path/script.js" defer></script>

Script Loading - XHR (Ajax)



var xhr = new XMLHttpRequest();
xhr.onload = function (responseScript) {
  // Execute response script
  eval(responseScript);
};
xhr.open('GET', 'URL');
xhr.send();

Script Loading - Document.write



document.write('');

Script Loading - Head.appendChild



var head = document.getElementsByTagName('head')[0],
    script = document.createElement('script');
script.src = url;
head.appendChild(script);

Script Loading - Function Wrapping


Famous script loading API such as RequireJS, CommonJS use syntax like :


require(["path/script.js"], function() {
  // This function is called after path/script.js has loaded.
});
